Initialization

Oh, the joys of initialization. There are a few too many kinds of initialization in C++. Most of the time, the distinctions are meaningless, but sometimes the subtleties can bite you in confusing and surprising ways if you're not up to speed.

copy-initialization

This is what happens when you initialize a variable using =, such as int x = 5;. It is directly equivalent to creating a temporary from the one argument, and then copying or moving that- hence, the name is a bit of a misnomer, since it might also be a move-initialization. The reason this is confusing is because if you have a non-movable non-copyable type, you therefore cannot use this form. In addition, some people expect an assignment, since the = kinda indicates that. But there is none.

value-initialization

This is the simplest form of initialization. All primitive types have a sensible default value- usually 0 or their equivalent of it, which is what you get when they are value-initialized. Types with a default constructor are default constructed. Primitive temporaries are value-initialized (for example int()) but primitive lvalues like int i; are not.

default-initialization

Effectively the same as value-initialization, but more fun, because primitive types are left uninitialized- reading them is UB. This is what happens when you do int i;. The upshot of this is that for all types except primitives, T t; is constructed properly and safe to go with.

direct-initialization

What happens when you call the constructor directly, through int i(5);. There are no semantic surprises or pitfalls here, but you can have the joys of the Most Vexing Parse.
